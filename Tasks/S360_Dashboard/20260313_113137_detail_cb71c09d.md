## Summary

This KPI requires Azure SignalR Service to disable local authentication on 15 Cognitive Services accounts that are still configured with API-key or other local-auth access. These accounts must be migrated to identity-based access first, then local auth must be turned off before the due date.

Using the new owner-inference rule chain, the current evidence supports likely owner annotations for 11 of the 15 resources. Four resources remain intentionally blank because the available signals either did not match the team alias map or were not available from the Azure metadata fallback.

## What This Task Is

The KPI is `[SFI-ID4.3.1] Cognitive Services - Safe Secrets Standard`. Its goal is to eliminate API keys and other local authentication methods on Cognitive Services resources and replace them with managed identities or Entra ID-based authentication.

For Azure SignalR Service, the S360 payload shows 15 affected `microsoft.cognitiveservices/accounts` resources, all in `Public` cloud and all tagged as `microsoft.cognitiveservices/accounts~nonprod`. Every flagged resource currently reports `issueType = resource local auth enabled`, which means the immediate non-compliant condition is that local authentication is still allowed on the account.

The required end state is:

1. workloads using these accounts must authenticate with Entra ID or managed identity instead of keys;
2. local authentication must be disabled on each Cognitive Services account;
3. the resulting resource configuration must be validated so callers still work after the auth change.

## Concrete Work Items

1. Inventory every caller of the 15 affected Cognitive Services accounts.
   Confirm which internal tools, agents, scripts, tests, or services still use keys or other local-auth paths.

2. Migrate callers to identity-based authentication.
   Replace API-key or local-auth usage with managed identity or Entra ID-based authentication before changing the resource setting.

3. Disable local auth on each Cognitive Services account.
   After callers are migrated, update each account so local authentication is disabled.

4. Validate end-to-end behavior after the auth change.
   Confirm each workload still functions once key-based access is removed.

5. Track remediation progress against the current ETA.
   Most items carry a service ETA of `2026-03-27` authored by `jixin`, so the remaining work should be aligned to that date and the final SLA date of `2026-03-31`.

6. Route investigation using the inferred owners where available.
  Start with the metadata-backed or high-confidence name-backed owners above, and separately triage the four resources that still have no matching owner in the current alias map.

## Prioritized Execution Order

1. Start with shared resource groups such as `pmagent`, where multiple Cognitive Services accounts are affected.
2. For each account, identify the current callers and whether they still depend on keys.
3. Migrate those callers to managed identity or Entra auth.
4. Disable local auth on the account.
5. Validate the workload and close any remaining dependency gaps.
6. Update status / ETA in S360 if any item cannot be completed by the current ETA or due date.
